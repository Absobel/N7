3 Mini-projet IDM

Le Mini-Projet (MP) consiste à construire une chaîne de vérification pour des modèles de processus. Il s’appuie sur les TP. Le but est de compléter les TP en ajoutant une extension : les ressources.

3.1 Principe de la transformation d’un modèle de processus en réseau de Petri

Le principe de la transformation d’un modèle SimplePDL en Réseau de Pétri est le suivant (voir correction du TD 1) :

  • Un élément Process devient un élément PetriNet.
  • Une WorkDefinition devient 4 places ready, started, running et finished et deux transitions start et finish. Il faut bien sûr avoir des noms différents pour les places et les transitions, on peut donc les préfixer par le nom de la WorkDefinition.

    Voici dans la syntaxe de Tina, le bout de réseau de Pétri correspondant à une WorkDefinition.

    pl XXX_ready (1)
tr XXX_start XXX_ready -> XXX_running XXX_started
tr XXX_finish XXX_running -> XXX_finished
  • Une WorkSequence devient un read arc entre une place de l’activité précédente (started ou finished) et une transition de l’activité cible (start ou finish).

Un processus se termine si toutes ses activités se terminent. La propriété correspondante peut s’écrire ainsi pour l’exemple du sujet : 

op finished = Conception_finished /\ RedactionTests_finished /\ RedactionDoc_finished /\ Developpement_finished;

[] (finished => dead);
[] <> dead ;
[] (dead => finished);
- <> finished;

Il faut noter que la première ligne dépend du processus ! Cette ligne référence la place finished de toutes les activités du processus.


Concernant les invariants qui permettent de valider la transformation, il s’agit de propriété vraies sur les processus qui devront donc aussi être vraies sur le réseau de Petri correspondant. Par exemple, les états ready, running et finished d’une activité sont exclusif. 

[] wd_ready + wd_running + wd_finished = 1;

Ceci doit bien sûr est déclinée pour chaque activité du processus en remplaçant wd par le nom de l’activité considérée.

3.4 Correctifs / Compléments

  1. Pourquoi vérifier la propriété - <> finished plutôt que <> finished ?

    La seconde permet de savoir si le processus termine (le model checker répond true). Mais on ne sait pas quel est le scénario qui permet de finir.

    Si le processus peut finir, la première propriété sera fausse et le model checker exhibera un contre-exemple qui correspond à un scénario qui correspond à la terminaison du processus.

6.2 Vérifier la version de Java : Au plus JDK 11

Pour savoir quelle version du JDK est utilisée par Eclipse, on peut faire Window > Preferences puis choisir Java / Installed JREs. Ce doit être une version 11 au plus. Attention, il ne faut que ce soit une version supérieure même si le mode compatibilité 11 est activé !

6.3 EMF/Java : Je n’arrive pas exécuter SimplePDLCreator.java

On peut déplier dans le "Project Explorer" la ressource SimplePDLCreator.java pour faire apparaître la méthode ​main. Sur cette méthode ​main, on peut faire clic droit, run as / Java application.

6.4 Comment tester plus rapidement ma transformation Acceleo : déployer un greffon est long.

Pour tester une transformation dans l’eclipse de développement on peut faire « run Configuations » (depuis le menu contextuel « run as » par exemple). Il faut alors double-cliquer sur Acceleo Application et renseigner les champs de l’assistant qui apparaît (voir cette capture d’écran) : le projet (qui contient la transformation), la transformation elle-même (cliquer sur search en fin de ligne listera les transformations disponibles), le modèle sur lequel appliquer la transformation (un modèle, pas un métamodèle !) et le dossier dans lequel engendrer les fichiers produits par la transformation (src-gen par exemple).

On peut ensuite rapidement relancer la transformation en utilisant le bouton vert d’exécution (lecture) sur la barre de bouton. On peut choisir l’exécution à effectuer en utilisant la petite flèche à la droite de ce bouton vert.

6.5 Acceleo ne fonctionne plus depuis que j’ai utilisé les genmodel et engrendré les éditeurs.

Il faut modifier le fichier .java qui correspond à la transformation Acceleo (par exemple ToPDL1.java pour toPDL1.mtl). Plus précisément, dans la méthode ​registerPackages​(​ResourceSet​), il faut :

  1. Ajouter NOT après @generated dans le commentaire de documentation (pour éviter que les modifications apportées ne soient écrasées par la prochaine génération: voir le premier commentaire de cette méthode).
  2. Ajouter l’instruction qui apparaît dans le deuxième commentaire. Il faut bien sûr l’adapter en fonction du métamodèle considéré.
  3. Si des erreurs apparaissent (ce sera certainement le cas car le SimplePDLPackage ou équivalent ne sera pas connu du projet), il faudra choisir parmi les corrections proposées par Eclipse : "Fix Project Setup".

Voici un exemple minimal du résultat attendu avec SimplePDL ‍:

/**
 * @generated NOT
 */
public void registerPackages(ResourceSet resourceSet) {
    super.registerPackages(resourceSet);

    if (!isInWorkspace(SimplepdlPackage.class)) {
        resourceSet.getPackageRegistry().put(SimplepdlPackage.eNS_URI, SimplepdlPackage.eINSTANCE);
    }
}

6.6 Sirius : le viewpoint n’apparaît pas dans la liste des viewpoint disponibles pour un projet avec un modèle dedans

Diagnostic : le registre des viewpoints ne se met pas bien à jour

Solution : retirer la nature « Modeling » puis la remettre pour forcer à mettre à jour : clic-droit sur le projet > Configure > Remove modeling Project Nature, puis clic-droit sur le projet > Configure > convert to Modeling Project.

6.7 Sirius : simplepdl:Process EClass cannot be found

Diagnostic : il manque les dépendances aux greffons simplepdl pour le projet design.

Solution : ouvrir le puglin.xml du projet .design, onglet "Dependencies", rajouter les trois plugins (simplepdl, simplepdl.edit, simplepdl.editor), faire Project > Clean puis redémarrer Eclipse (c’est suffisant pour corriger le problème normalement).
